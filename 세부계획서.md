# 지하철 도착정보 상단 패널 서비스 세부계획서

## 1. 문서 목적
이 문서는 "앱을 열지 않아도 휴대폰 상단 알림 패널을 내려 즐겨찾기 역의 지하철 도착정보를 빠르게 확인하는 서비스"를 실제로 만들기 위한 기준 문서다.

핵심은 단순하다.

- 일반 지하철 앱: 앱을 열고, 역을 검색하고, 화면을 봐야 한다.
- 이 서비스: 사용자가 화면을 위에서 아래로 내려 상단 알림 패널만 확인해도, 즐겨찾기 역의 실시간 도착정보를 바로 본다.

즉, "앱 실행"이 아니라 "즉시 확인"이 목적이다.

## 2. 서비스 목적
### 2.1 해결하려는 문제
출근길이나 이동 중 사용자는 지하철 도착정보를 자주 확인하지만, 기존 방식은 동작 수가 많다.

- 앱 실행
- 역 검색 또는 즐겨찾기 선택
- 도착정보 화면 진입
- 현재 정보를 다시 확인

이 과정은 짧아 보여도 반복되면 피로하다. 특히 잠금 해제 직후, 다른 앱 사용 중, 길 찾는 중에는 더 귀찮다.

### 2.2 목표
사용자가 앱을 직접 열지 않고도, 휴대폰 화면 상단의 Android 알림 패널을 내렸을 때 즐겨찾기 역의 실시간 도착정보를 확인하게 만든다.

### 2.3 핵심 가치
- 접근 속도: 최소 동작으로 확인
- 반복 편의성: 하루에도 여러 번 자연스럽게 재확인 가능
- 개인화: 사용자가 저장한 역 중심으로 정보 제공
- 실용성: "지금 몇 분 남았는가"를 바로 보여줌

## 3. 대표 사용자 시나리오
### 시나리오 A: 출근 직전
- 사용자는 집에서 나가기 전 휴대폰 화면을 아래로 내려 상단 알림 패널을 확인한다.
- 즐겨찾기 역 `강남역 2호선`의 실시간 도착정보가 바로 보인다.
- "2분 후 도착", "전역 출발" 같은 정보로 바로 이동 여부를 판단한다.

### 시나리오 B: 이동 중 재확인
- 사용자는 지도 앱이나 메신저를 쓰는 중이다.
- 앱을 끄고 다시 찾을 필요 없이 상단 알림 패널만 내려서 도착정보를 다시 본다.

### 시나리오 C: 자주 가는 역 관리
- 사용자는 자주 쓰는 역 1~3개를 즐겨찾기로 저장한다.
- 시스템 영역에는 현재 선택된 대표 역 정보가 노출된다.
- 사용자는 좌우 넘김 동작으로 즐겨찾기 역 사이를 이동하며 확인한다.
- 하단의 페이지 표시값으로 현재 몇 번째 즐겨찾기 역을 보고 있는지 인지한다.
- 사용자는 앱 검색 화면 우측 상단 버튼으로 즐겨찾기 관리 화면에 진입한다.
- 즐겨찾기별로 필요하면 특정 행선지 필터를 추가로 설정한다.

## 4. 타겟 사용자
- 출퇴근으로 특정 역을 반복 이용하는 사용자
- 앱을 열고 검색하는 과정을 번거롭게 느끼는 사용자
- 실시간성보다도 "빨리 보는 경험"을 더 중요하게 생각하는 사용자

## 5. 핵심 기능 정의
### 5.1 반드시 있어야 하는 기능
- 즐겨찾기 역 등록/삭제
- 즐겨찾기 역 순서 지정
- 역 검색 기반 즐겨찾기 추가
- 검색 결과에 `상행/하행` 방향 정보 표시
- 즐겨찾기 관리 화면
- 즐겨찾기별 행선지 필터 설정
- 역별 실시간 도착정보 조회
- 상단 알림 패널에서 도착정보 노출
- 즐겨찾기 역 좌우 넘김 전환
- 상단 패널 내 새로고침 버튼
- 마지막 갱신 시각 표시
- 오류 또는 데이터 없음 상태 표시

### 5.2 있으면 좋은 기능
- 복수 역 순환 표시
- 특정 시간대 자동 고정 역 설정
- 급행/일반 구분 강조
- 혼잡도, 첫차/막차, 출구 정보 연계

## 6. 플랫폼별 현실 정리
여기가 중요하다. "상단 패널을 내렸을 때 실시간 도착정보가 보이는 경험"은 Android와 iOS에서 완전히 같은 방식으로 구현되지 않는다.

### 6.1 Android
비교적 유리하다.

- 사용자가 상단바를 내려 확인하는 Android 알림 패널을 1차 목표 UX로 설계
- 알림 패널 안에서 이전/다음, 새로고침 같은 버튼형 상호작용을 제공하는 구조를 목표로 한다.
- 앱이 종료된 뒤에도 패널을 유지하려면 포그라운드 서비스 기반 구성이 사실상 필요할 가능성이 높다.
- 위젯, 빠른 설정 타일, 잠금화면 연계는 후속 확장 수단으로만 검토한다.

즉, 사용자가 화면을 내렸을 때 확인하는 경험과 가장 잘 맞는다.

### 6.2 iOS
제약이 크다.

- Android처럼 임의의 상단 상태 영역에 지속 정보 노출은 제한적
- 위젯, Live Activity, 알림 중심으로 우회 설계 필요
- 백그라운드 실시간 갱신은 정책 제약이 큼

즉, iOS는 "완전히 같은 UX"가 아니라 "비슷한 목적을 달성하는 대체 UX"로 설계해야 한다.

### 6.3 결론
- 제품 1차 목표는 Android 알림 패널 중심의 핵심 경험을 완성하는 것이다.
- iOS는 MVP 범위에서 구현 대상이 아니라, Android MVP 검증 이후 별도 단계로 설계한다.

즉, 기획과 데이터 구조는 확장 가능하게 잡되, 실제 제품 개발과 검증은 Android부터 진행하는 것이 맞다.

## 7. 데이터 소스 전략
### 7.1 1순위 API
서울 열린데이터광장 `서울시 지하철 실시간 도착정보` Open API를 기본 데이터 소스로 사용한다.

이유:

- 서비스가 보여주려는 값이 "도착정보" 그 자체다.
- 위치정보를 계산해서 도착예정을 추정하는 것보다 직접적이다.
- 구현 난이도와 오차 가능성을 줄인다.

참고 페이지:

- https://data.seoul.go.kr/dataList/OA-12764/A/1/datasetView.do

### 7.2 2순위 보조 API
서울 열린데이터광장 `서울시 지하철 실시간 열차 위치정보` Open API는 보조 수단으로만 검토한다.

사용 목적:

- 도착정보 품질 검증
- 특정 상황에서 보조 설명값 제공
- 향후 열차 이동 시각화 기능 확장

참고 페이지:

- https://data.seoul.go.kr/dataList/OA-12601/A/1/datasetView.do

### 7.3 데이터 사용 시 주의점
- 서울시 외 구간 정보는 일부 미제공일 수 있다.
- 실시간 데이터는 생성 시각과 현재 시각 사이에 차이가 있을 수 있다.
- API 응답 지연, 누락, 일시 장애를 전제로 설계해야 한다.

즉, 앱은 "항상 정확한 초 단위 실시간"이 아니라 "실사용에 충분한 최신 정보"를 목표로 해야 한다.

### 7.4 응답 포맷 방침
서울 Open API는 XML이 아니라 JSON 형식으로 요청하고 수신하는 것을 기본 원칙으로 한다.

이유:

- React Native + Expo와 Workers에서 파싱 구조를 단순하게 유지하기 쉽다.
- Workers에서 응답 가공과 정규화가 쉽다.
- 타입 정의와 유지보수가 XML보다 단순하다.
- 실시간 데이터 처리에서 불필요한 파싱 복잡도를 줄일 수 있다.

즉, 샘플 테스트를 XML로 했더라도 실제 서비스 설계 기준은 `JSON 우선`이다.

## 8. 기술 선택
### 8.1 앱 프레임워크
React Native + Expo를 우선 선택한다.

이유:

- Android MVP를 빠르게 만들고, 이후 iOS 확장 시 재사용 가능한 코드베이스를 확보할 수 있음
- TypeScript 기반으로 API 타입과 화면 로직을 한 흐름으로 관리하기 쉬움
- Expo 기반으로 초기 셋업, 빌드, 배포 준비 속도가 빠름
- 검색, 즐겨찾기, 알림 같은 MVP 범위를 검증하기에 필요한 기능 구성이 충분함

단, 1차 구현은 Android 중심으로 진행하며, 시스템 영역 기능은 Android 네이티브 코드가 일부 필요할 수 있다. iOS 대응은 후속 단계에서 별도로 붙인다.

### 8.2 백엔드
Cloudflare Workers를 우선 사용한다.

역할:

- 서울 Open API 프록시
- JSON 응답 수신 및 표준 스키마로 재구성
- API 키 은닉
- 응답 가공/정규화
- 짧은 캐싱
- 장애 시 공통 에러 응답 제공

### 8.3 웹/랜딩
Cloudflare Pages는 소개 페이지, 정책 페이지, 다운로드 안내 페이지 용도로 사용한다.

중요:

- Cloudflare Pages만으로 모바일 앱 자체를 배포할 수는 없다.
- 실제 앱 배포는 App Store, Google Play가 담당한다.

## 9. 권장 아키텍처
### 9.1 구조
- 모바일 앱(React Native + Expo)
- 앱 내부 데이터 계층
- Cloudflare Workers API
- 서울 열린데이터 Open API

### 9.2 데이터 흐름
1. 사용자가 즐겨찾기 역을 저장한다.
2. 앱은 마지막으로 표시하던 즐겨찾기와 사용자가 정한 순서를 로컬에 저장한다.
3. 앱 또는 시스템 표시 모듈이 Workers API를 호출한다.
4. Workers가 서울 Open API를 호출하고 필요한 필드만 정규화한다.
5. 앱은 행선지, 도착예정시간, 갱신 상태를 상단 패널에 표시한다.
6. 실패 시 본문은 오류 상태로 바꾸고, 우측 상단에는 마지막 성공 수신 시각을 유지한다.

### 9.3 왜 프록시가 필요한가
- API 키를 앱에 직접 넣으면 노출 위험이 크다.
- 서울 Open API 응답 형식을 앱이 직접 다루면 유지보수가 힘들다.
- 캐싱과 재시도 정책을 서버에서 통합하는 편이 단순하다.

## 11. UX 원칙
### 11.1 상단 패널 정보는 짧아야 한다
길게 설명하면 실패다. 패널 안에서도 사용자가 한눈에 읽어야 한다.

예시:

- `강남역 2호선`
- `서초행 2분 20초`
- `성수행 5분 10초`

### 11.2 꼭 필요한 것만 보여준다
- 역명
- 노선
- 한 줄에 배치한 첫 번째와 두 번째 도착 열차 정보
- 각 열차의 최종 행선지와 남은 시간
- 우측 상단의 새로고침 버튼과 마지막 성공 수신 시각
- 현재 보고 있는 즐겨찾기 위치 표시

### 11.2.1 상단 패널 문구 표현 방식
- 첫 줄 좌측에는 `역명 + 노선명`을 텍스트로 표시한다.
- 첫 줄 우측에는 누를 수 있는 새로고침 버튼 `[↻]`과 `10초 전`, `1분 전` 같은 상대시간을 표시한다.
- 아직 한 번도 성공적으로 수신한 적이 없으면 우측 상단 시간 영역은 `-`로 표시한다.
- 갱신 중에는 새로고침 버튼만 `[...]` 형태로 바꾸고, 시간 텍스트는 유지한다.
- 본문 도착정보는 한 줄에 두 열차를 함께 표시한다.
- 각 열차 문구 형식은 `성수행 2분 20초`처럼 `최종 행선지 + 시간` 순서를 기본으로 한다.
- 두 열차 사이는 `|` 문자로 구분한다.
- 열차가 1대만 있으면 `성수행 2분 20초 | -` 형태로 표시한다.
- 행선지 필터가 있으면 필터에 맞는 열차만 표시하고, 조건에 맞는 열차가 없으면 `도착 정보 없음`으로 처리한다.
- 시간 표시는 서울시 도착정보 API의 `barvlDt` 값을 기준으로 한다. `barvlDt`는 남은 시간을 초 단위로 준다.
- 앱은 새 도착정보를 받을 때 `barvlDt`를 현재 남은 초 값으로 저장하고, 화면에서는 이 값을 1초마다 1씩 감소시키는 카운트다운 방식으로 표시한다.
- 분/초 표시는 `분 = Math.floor(barvlDt / 60)`, `초 = barvlDt % 60` 규칙으로 계산한다.
- 예: `barvlDt = 140`이면 `2분 20초`, 1초 뒤에는 `2분 19초`로 표시한다.
- `arvlMsg2`는 기본 시간 계산용 값이 아니라 예외 상태 문구 처리용으로만 사용한다.
- `barvlDt`가 0 이하이거나 숫자가 비정상이면 `arvlMsg2`를 보고 `전역 도착`, `도착`, `진입` 같은 상태 문구를 그대로 표시한다.
- `barvlDt`가 정상 숫자이면 `arvlMsg2`보다 `barvlDt`를 우선 사용한다.
- 내부 규칙으로 `곧 도착`을 계산할 때 기준은 `barvlDt` 10초 이하로 한다.
- 행선지 텍스트가 길면 글자 크기를 줄이지 않고 말줄임표 처리한다.

### 11.2.2 패널 카드 레이아웃 기준
패널 내부 도착정보 카드는 `가로로 길고 세로는 짧은 알림 패널 카드` 형태를 기준으로 설계한다.

```text
┌──────────────────────────────────────────────────────────────┐
│  강남역 2호선                                   [↻] 10초 전 │
│  성수행 2분 20초 | 신사행 전역 도착                         │
│                          [<]  1/3  [>]                      │
└──────────────────────────────────────────────────────────────┘
```

레이아웃 원칙:

- 첫 줄: `강남역 2호선`처럼 역명 뒤에 노선명을 텍스트로 표시한다.
- 첫 줄 우측: 새로고침 버튼과 마지막 성공 수신 기준 상대시간을 표시한다.
- 둘째 줄: 첫 번째, 두 번째 열차 정보를 한 줄에 함께 표시한다.
- 셋째 줄: 카드 하단 중앙에 이전/다음 전환용 버튼과 현재 위치 `1/3`을 표시한다.
- `<` 버튼을 누르면 이전 즐겨찾기 역 카드로 이동하고, `>` 버튼을 누르면 다음 즐겨찾기 역 카드로 이동한다.
- 좌우 이동은 끝에서 멈추지 않고 순환한다.
- 즐겨찾기가 1개뿐이면 좌우 버튼은 숨기고 `1/1`만 표시한다.
- 상단 패널 카드를 누르면 앱의 검색 화면으로 이동한다.
- 새로고침 버튼을 누르면 현재 보고 있는 즐겨찾기 1개만 갱신한다.

### 11.3 데이터가 이상할 때 숨기지 말고 상태를 보여준다
예:

- `정보 갱신 실패`
- `서울 외 구간 미지원`

원칙:

- 새 조회가 실패하면 마지막 성공 도착정보는 유지하지 않고 본문을 `정보 갱신 실패`로 바꾼다.
- 실패 상태에서도 우측 상단 시간은 마지막 성공 수신 시각 기준으로 유지한다.
- `도착 정보 없음`일 때는 도착정보 두 줄 대신 `도착 정보 없음` 한 줄만 표시한다.
- `도착 정보 없음` 상태에서도 좌우 버튼과 위치 표시는 유지한다.
- 아직 즐겨찾기가 하나도 없으면 `즐겨찾기를 추가하세요`를 표시한다.

### 11.4 역 검색과 노선 식별은 한눈에 보여야 한다
- 사용자는 역 검색으로 즐겨찾기 대상을 찾을 수 있어야 한다.
- 앱 실행 첫 화면은 검색 화면이다.
- 검색 화면 우측 상단 버튼으로 즐겨찾기 관리 화면에 진입한다.
- 검색은 역 이름 기준으로 수행한다.
- 검색은 2글자 이상 입력했을 때부터 즉시 시작한다.
- 검색창 우측에는 입력값을 지우는 `X` 버튼을 둔다.
- 검색 결과는 `역명 + 노선 + 상행/하행`이 각각 별도 항목으로 나온다.
- 사용자는 원하는 검색 결과 항목의 행 전체를 눌러 즐겨찾기 추가를 시작한다.
- 즐겨찾기 추가 전에는 확인 팝업을 한 번 보여준다.
- 이미 추가된 항목은 다시 눌러도 중복 저장되지 않는다.
- 이미 추가된 항목을 누르면 `이미 추가된 즐겨찾기입니다`를 표시한다.
- 즐겨찾기가 3개인 상태에서 새 항목을 추가하려 하면 `즐겨찾기는 최대 3개까지 가능합니다`를 표시한다.
- 검색 결과는 일치도 높은 순으로 전부 표시한다.
- 검색 결과가 없으면 `검색 결과가 없습니다`를 표시한다.
- 검색창 기본 안내 문구는 `역 이름을 검색하세요`로 한다.

### 11.4.1 즐겨찾기 관리 화면
- 검색 화면과 즐겨찾기 관리 화면은 분리한다.
- 즐겨찾기 관리 화면에서는 최대 3개의 즐겨찾기를 목록으로 보여준다.
- 즐겨찾기 항목은 `역명 + 노선 + 상행/하행` 형식으로 표시한다.
- 행선지 필터가 있는 항목은 `역명 + 노선 + 상행/하행 + 행선지` 형식으로 추가 표시한다.
- 즐겨찾기 항목은 드래그로 순서를 바꾸며, 드래그 결과는 즉시 저장한다.
- 휴지통 버튼을 누르면 즉시 삭제한다.
- 행 전체를 누르면 즐겨찾기 상세 설정 화면으로 이동한다.
- 뒤로 가기 시 검색 화면으로 돌아간다.

### 11.4.2 즐겨찾기 상세 설정 화면
- 즐겨찾기 상세 설정 화면에서는 행선지 필터만 수정한다.
- 필터 목록은 `필터 없음`, `성수행`, `신사행` 같은 값으로 구성한다.
- 기본값은 `필터 없음`이다.
- 저장 후에는 `저장되었습니다`를 표시하고 즐겨찾기 관리 화면으로 돌아간다.

### 11.5 대표 표시 역 전환 방식
- 대표 표시 역은 자동 순환이 아니라 사용자가 좌우로 넘겨 직접 전환하는 방식을 기본으로 한다.
- 좌우 넘김은 노선 그룹이 아니라 `즐겨찾기한 역` 단위로 한 칸씩 이동한다.
- 예: `2호선 강남역 -> 7호선 보라매역 -> 1호선 회기역`
- 패널 내부 UI는 여러 카드가 동시에 보이는 캐러셀이 아니라, `카드 1개를 좌우로 넘기는 구조`를 기본으로 한다.
- 하단에는 `1/3` 형태로 현재 위치를 보여준다.
- 즐겨찾기 순서는 사용자가 드래그로 바꿀 수 있으며, 좌우 넘김은 그 순서를 따른다.
- 대표 표시 역은 마지막으로 상단 패널에서 보고 있던 항목을 기억한다.
- 좌우 이동은 끝에 도달하면 반대편으로 이어져 순환한다.
- 좌우 버튼으로 다른 즐겨찾기로 넘기면 즉시 해당 항목을 새로 조회한다.

## 12. MVP 범위
### 12.1 MVP에 포함
- 단일 사용자 로컬 기반 앱
- 즐겨찾기 역 1~3개 저장
- 앱 실행 시 검색 화면 진입
- 검색 화면과 즐겨찾기 관리 화면 분리
- 검색 결과 `역명 + 노선 + 상행/하행` 표시
- 검색 결과 선택 시 확인 팝업 후 즐겨찾기 추가
- 대표 역 1개 패널 표시
- 즐겨찾기 역 좌우 넘김 전환
- 패널 내부 카드 1장씩 전환하는 구조
- 현재 위치를 나타내는 `1/3` UI
- 즐겨찾기 1개일 때 `1/1` 표시
- 수동 새로고침
- 수동 새로고침은 현재 보고 있는 즐겨찾기 1개만 갱신
- 화면이 켜져 있을 때만 10초 자동 갱신
- 화면이 다시 켜지면 자동 갱신 재시작
- 자동 갱신은 현재 보고 있는 즐겨찾기 1개만 수행
- Cloudflare Workers 프록시
- Android 전용 MVP 출시
- iOS 확장을 고려한 데이터 모델 및 API 구조 유지
- 즐겨찾기 삭제
- 즐겨찾기 드래그 순서 변경
- 즐겨찾기 행선지 필터 설정
- 앱 종료 후에도 유지되는 Android 알림 패널
- 상단 패널 클릭 시 검색 화면 이동

### 12.2 MVP에서 제외
- 회원가입/로그인
- 서버 DB 동기화
- 열차 이동 지도 시각화
- 다국어
- iOS 앱 구현 및 배포
- Apple 완전 동일 경험 보장
- 고급 추천 기능

### 12.3 MVP 성공 기준
- 사용자가 앱을 열지 않고도 3초 안에 즐겨찾기 역 도착정보를 확인할 수 있다.
- 즐겨찾기 설정이 1분 안에 끝난다.
- 데이터 없음/실패 시 잘못된 정보 대신 상태를 명확히 표시한다.
- 앱 내부에서는 도착정보 상세 화면 없이 검색, 즐겨찾기 관리, 필터 설정만으로 목적을 달성할 수 있다.

- 검색 결과 `역 + 노선 + 상행/하행` 개별 항목 UI
- 즐겨찾기 삭제 및 드래그 정렬 UI
- 검색 결과 선택 시 확인 팝업 후 저장 흐름
- 로컬 저장소 연동

### 4단계: Android 시스템 영역 연동
- Android 알림 패널 지속 노출 방식 구현

- 새로고침 버튼 연결

- 실제 사용 시나리오 기준 QA
- 내부 테스트 배포 준비

- 유지 비용 대비 iOS 확장 우선순위 판단

## 14. 운영 정책 초안
### 14.1 호출 주기
- 기본 원칙은 화면이 켜져 있고 사용자가 확인 가능한 순간에 우선 갱신하는 것이다.
- 화면이 켜져 있을 때는 현재 보고 있는 즐겨찾기 1개만 10초 주기로 갱신한다.
- 화면이 꺼져 있으면 자동 갱신을 중지한다.
- 화면이 다시 켜지면 현재 보고 있는 즐겨찾기 기준으로 자동 갱신을 재시작한다.
- 좌우 버튼으로 다른 즐겨찾기로 이동하면 즉시 해당 항목을 조회한다.
- 수동 새로고침도 현재 보고 있는 즐겨찾기 1개만 대상으로 한다.

### 14.2 캐싱
- Workers에서 초단기 캐싱 적용
- 동일 역 중복 호출 완화
- 실패 시 마지막 성공 시각은 유지하되 본문 도착정보는 에러 상태로 교체

### 14.3 장애 대응
- 서울 Open API 실패 시 즉시 재시도 남발 금지
- 사용자에게 `도착 정보 없음`, `정보 갱신 실패`, `마지막 성공 수신 시각` 표시
- 모니터링 로그에 역명, 시각, 실패 유형 기록

## 15. 주요 위험 요소와 대응
### 위험 1: iOS 제약으로 Android와 같은 UX 불가
대응:

- 처음부터 플랫폼 공통 UX를 강제하지 않는다.
- Android 중심 MVP로 간다.
- iOS는 대체 경험으로 정의한다.

### 위험 2: 실시간 데이터 품질 흔들림
대응:

- 도착정보 API를 기본 사용
- 생성 시각 기준 `stale` 판단 로직 추가
- 잘못된 확정값처럼 보이게 표시하지 않는다.

### 위험 3: 배터리 소모
대응:

- 화면 켜짐 상태에서만 10초 갱신
- 현재 보고 있는 즐겨찾기 1개만 조회
- 필요 시 사용자 설정 제공
- 백그라운드 동작 범위를 최소화

### 위험 4: 시스템 정책 변경
대응:

- 표시 기능을 알림, 위젯, 앱 홈 화면으로 분리 설계
- 특정 OS 기능에 과도하게 종속되지 않도록 추상화

### 위험 5: 상단 패널에서의 스와이프와 모션 표현 제약
대응:

- 좌우 넘김과 모션형 표현은 목표 UX로 두되, OS가 허용하는 범위 안에서만 구현한다.
- 상태표시줄 자체 제어가 불가능하면 알림 패널 카드, 위젯, 시스템 카드 등 대체 가능한 패널 UX로 전환한다.
- `모션이 반드시 있어야 서비스가 성립한다`는 구조로 설계하지 않는다.

## 16. 완료 기준
- Android에서 즐겨찾기 역 도착정보가 알림 패널에서 확인된다.
- Cloudflare Workers를 통해 서울 API를 안정적으로 호출한다.
- 데이터 없음, 오류, 마지막 갱신 시각이 구분되어 표시된다.
- Android MVP 기준으로 핵심 사용자 흐름이 실제 기기에서 검증된다.
- 비전공자도 "왜 이 구조인지" 문서만 읽고 이해할 수 있다.
