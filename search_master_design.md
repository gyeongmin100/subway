# 검색용 마스터 데이터 설계

## 목표
- 검색 단계에서 `역명 + 노선 + 행선지`를 정확하게 보여줄 수 있는 기준 데이터를 정의한다.
- 분기선, 지선, 순환선이 있는 서울권 전체 노선에서도 사용자가 잘못된 방향을 선택하지 않도록 한다.

## 적용 범위
- 수도권 지하철 전체 검색 데이터
- 즐겨찾기 등록용 기준 데이터
- 실시간 도착정보 API 응답과 매칭 가능한 검색 키 설계

## 핵심 원칙
- 검색 결과는 `역명 + 노선 + 행선지` 단위로 보여준다.
- 같은 역에서 가능한 행선지는 모두 보여준다.
- 즐겨찾기 저장 단위도 `역명 + 노선 + 행선지`로 통일한다.
- 공식 데이터 우선, 부족한 구간만 보정 테이블로 덧붙인다.

## 사용 데이터 소스
### 1. 기본 역/노선 마스터
- `국토교통부_도시철도 전체노선_20241211.csv`
- 용도:
  - 노선 목록
  - 역 목록
  - 노선 내 순서
  - 수도권 전체 범위 확보

### 2. 역 코드 보강
- `서울교통공사_노선별 지하철역 정보.json`
- 용도:
  - 역명
  - 노선명
  - 전철역코드
  - 외부코드

### 3. 실시간 도착정보 원천
- 서울시 `OA-12764` 실시간 도착정보 API
- 용도:
  - 실제 표시 행선지
  - 상하행/내외선 문구
  - 도착 정보와 검색 데이터 연결 검증

## 검색 마스터 데이터 구조
- `stationName`
- `lineName`
- `operatorName`
- `stationCode`
- `externalCode`
- `branchKey`
- `destinationName`
- `displayLabel`
- `searchTokens`
- `isActive`

## 필드 정의
### stationName
- 사용자에게 보여줄 역명
- 예: `강남`

### lineName
- 사용자에게 보여줄 노선명
- 예: `2호선`, `신분당선`

### operatorName
- 운영기관명
- 예: `서울교통공사`, `코레일`

### stationCode
- 역 식별 코드
- 가능하면 공식 코드 사용

### externalCode
- 외부코드
- 서울교통공사 데이터가 있는 경우만 채움

### branchKey
- 분기/지선/순환 구간 식별용 내부 키
- 예:
  - `2호선-본선-외선`
  - `2호선-성수지선`
  - `5호선-마천지선`
  - `5호선-하남연장`

### destinationName
- 검색 결과에 표시할 행선지
- 예: `성수행`, `광교행`, `마천행`

### displayLabel
- 최종 검색 표시 문자열
- 예: `강남역 2호선 성수행`

### searchTokens
- 검색용 보조 토큰 배열
- 예:
  - `강남`
  - `강남역`
  - `2호선`
  - `성수`
  - `성수행`

### isActive
- 현재 서비스에서 노출할지 여부
- 임시 제외 구간 제어용

## branchKey 설계 원칙
- 단순 노선은 `노선명-행선지` 수준으로도 가능하다.
- 분기 노선은 선로 단위로 별도 키를 만든다.
- 하나의 역이 여러 행선지를 가질 수 있으면 행 단위를 분리한다.

## 보정 테이블이 필요한 이유
- 공식 데이터만으로는 분기선과 순환선을 완전하게 표현하기 어렵다.
- 같은 `노선명` 안에 여러 갈래가 섞여 있는 구간이 있다.
- 검색 단계부터 정확한 방향을 보여주려면 내부 보정 규칙이 필요하다.

## 보정 테이블 구조
- `lineName`
- `stationName`
- `branchKey`
- `destinationName`
- `priority`
- `enabled`
- `note`

## 보정 테이블 역할
- 특정 역에서 가능한 행선지 목록 정의
- 같은 역/노선 조합의 중복 검색 결과를 구분
- 공식 데이터 순서만으로 해결되지 않는 분기 구간 보정
- 실시간 API 응답 문구와 검색 결과 문구를 일치시키기 위한 기준 제공

## 검색 결과 생성 규칙
1. 기본 역/노선 마스터를 읽는다.
2. 노선별 분기 규칙을 적용해 `branchKey`를 만든다.
3. 각 역에서 가능한 행선지 목록을 생성한다.
4. `역명 + 노선 + 행선지` 단위로 검색 결과 행을 만든다.
5. 보정 테이블에 있는 항목으로 최종 값을 덮어쓴다.
6. `displayLabel`과 `searchTokens`를 생성한다.

## 실시간 API 매칭 규칙
- 실시간 도착정보의 `trainLineNm`, `arvlMsg3`, `updnLine`를 우선 사용한다.
- 검색 마스터의 `destinationName`은 실시간 API 문구와 최대한 동일하게 맞춘다.
- 저장된 즐겨찾기와 실시간 응답을 비교할 때는 완전 일치보다 정규화 비교를 우선한다.

## 정규화 규칙
- `성수행`과 `성수 방면`은 내부 비교용 정규화 값으로 통일한다.
- 괄호, 공백, 특수문자 차이는 비교 전에 제거한다.
- `역` 접미사는 검색 토큰에는 포함하되 내부 식별 키에서는 제거한다.

## MVP 구현 순서
1. 공식 데이터 기반 기본 마스터 생성
2. 분기/지선/순환선 목록 확정
3. 보정 테이블 초안 작성
4. 검색 결과 샘플 검증
5. 실시간 API 응답과 행선지 매칭 검증
6. 앱 검색 UI 연결

## 완료 조건
- 검색창에서 사용자가 `역명`을 입력하면 정확한 `역명 + 노선 + 행선지` 결과가 나온다.
- 갈라지는 노선에서도 잘못된 방향 결과가 섞이지 않는다.
- 즐겨찾기 저장값이 실시간 도착정보와 안정적으로 매칭된다.
